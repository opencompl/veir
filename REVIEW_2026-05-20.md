**Code says otherwise**: `Veir/IR/Basic.lean:103-105`:
```lean
inductive ValuePtr where
  | opResult (ptr : OpResultPtr)
  | blockArgument (ptr : BlockArgumentPtr)

instance : Coe BlockArgumentPtr ValuePtr where
  coe ptr := ValuePtr.blockArgument ptr
```

## 4. Hidden findings (not surfaced by docs)

### MEDIUM ‚Äî LLZK tests skip VEIR_ROUNDTRIP
Upstream PR #605 added a `VEIR_ROUNDTRIP` lit substitution that
runs `veir-opt %s | veir-opt | filecheck %s` ‚Äî a parse-print-parse
double round-trip, stronger than our `veir-opt %s | filecheck %s`.

Twenty upstream tests adopted it. **Zero LLZK tests use it.** Our
identity tests would silently miss any second-round-trip drift.
Manual spot-check on Felt/Bool/Function shows they are stable
under double round-trip, but the suite does not enforce it.

### LOW ‚Äî `resolveFlatSymbol` is API-only (no consumers)
`Veir/IR/SymbolTable.lean`'s `resolveFlatSymbol` / `resolveSymbol`
have no callers in `Veir/`, `Test/`, `UnitTest/`, or `scripts/`.
This is consistent with the doc ("currently a forward-compatibility
stub") but plan.md row 39 reads as if F.5 consumes the walker ‚Äî
F.5 does not. The walker is genuinely dead code until G.1
introduces `function.call`.

### LOW ‚Äî Latent UTF-8 panic in resolveFlatSymbol
`Veir/IR/SymbolTable.lean:81`:
```lean
let storedRef := "@" ++ String.fromUTF8! name.value
```
`fromUTF8!` panics on invalid UTF-8. StringAttrs from the parser
*should* be valid UTF-8 (they came from string literals), but
this is a latent panic. Recommend `String.fromUTF8?` with a
`continue` on `none`.

### LOW ‚Äî Terminator classifier won't trip Gotcha 1
`Veir/IR/Terminators.lean:34` has a default `| _ => false` arm.
This means a future dialect port that adds a terminator op will
silently fail to classify it (no compile error from Gotcha 1).
This is fine for the wildcard-style hygiene of the new file but
should be added as a porting-notes pattern: "when adding a
terminator op, also add an arm to `OpCode.isTerminator`."

### LOW ‚Äî Upstream linter warnings in Veir/Verifier.lean
`Veir/Verifier.lean:2107` (and again at ~2120) flag three unused
simp args (`throw`, `throwThe`, `MonadExceptOf.throw`) in the
post-PR-510 `arith_constant` / `arith_addi` theorems, made unused
by the PR #587/#588 `ParserError` refactor. **Not our bug** ‚Äî
upstream cleanup. Surfaces on every `lake exec`.

### INFO ‚Äî Dataflow Framework absent from LLZK code
Upstream PR #264 added `Veir/Analysis/DataFlowFramework.lean` and
`Veir/Analysis/DataFlow/` to VEIR. No LLZK dialect registers a
fact with it. Opportunity rather than gap ‚Äî future work on
constraint propagation, witness/constraint-gen analysis, etc.
